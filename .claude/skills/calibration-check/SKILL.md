# Mull philosopher calibration check

This skill audits `lib/philosophers.ts`. It exists because the Wave 2
corpus (394 entries) was generated from tradition baselines plus
per-entry overrides — a process that's authentically positioning,
not default-5, but not individually stress-tested. Future Wave 3+
additions will use the same workflow and need the same audit.

## When to use

- The user just added philosophers and asks "did that work?"
- The user asks whether a specific entry is correctly placed
- The user asks for a calibration / sanity / sense check
- The user mentions "Hammer is underrepresented" or any archetype
  count anomaly — this skill produces the canonical distribution
- After running `node scripts/gen-philosophers.mjs --apply`

## What the skill does, in order

### 1. Run the mechanical check

```bash
node scripts/check-philosopher-calibration.mjs
```

This produces a markdown report at `scripts/calibration-report.md`
with:

- **Mean / median top-1 nearest-kin similarity.** Low mean means the
  corpus is sparse; high mean means densely packed. Mull's corpus is
  typically ~0.99, which is fine — vectors are 16-D and all positive.
- **Most-isolated entries** (low top-1 similarity). Each comes with
  its top-5 nearest kin. Isolation is sometimes genuine (Buddha is
  far from anyone in vector space), sometimes a calibration bug.
- **Most-ambiguous entries** (small archetype margin). These sit
  between two archetypes. Honest for paradoxical thinkers; red flag
  for everyone else.
- **Archetype distribution.** Counts per archetype. Sharp imbalance
  (e.g. one archetype with < 5 entries) is worth investigating.

### 2. Semantic review of flagged entries

For each entry in the top of the isolation OR ambiguity list, read:

- The entry's `keyIdea` field (one-line summary of their philosophy)
- Their `archetypeName` (what the vector classified them as)
- Their top-5 nearest kin from the mechanical report

Then ask: does the keyIdea match the archetype, and do the nearest
kin make sense as philosophical peers?

Patterns that mean it's FINE:

- Top kin are tradition-mates (Nagarjuna near Vasubandhu — both
  Madhyamaka Buddhists)
- Top kin are a teacher/student or critic pairing (Plato near
  Diotima; Marx near Angela Davis)
- Top kin are temporally distant but conceptually identical
  (Heraclitus near Bergson — both process philosophers)

Patterns that mean the vector PROBABLY NEEDS WORK:

- Top kin share surface signals but not deep stance (Ayn Rand near
  Mary Wollstonecraft — both individualist/sovereign-self but
  philosophically incompatible)
- Archetype classification doesn't match the keyIdea ("amor fati"
  classified as Cartographer instead of Hammer)
- Top kin span totally unrelated traditions with no plausible bridge

### 3. Produce a verdict

After reviewing the flagged entries, write a short summary covering:

- Total entries reviewed
- Entries judged FINE (with one-line reason each)
- Entries judged SUSPICIOUS (with one-line reason + suggested fix)
- Whether the user should adjust gen-philosophers.mjs tradition
  baselines, add per-entry overrides, or leave it

If suggesting a vector adjustment, propose specific dim changes
(e.g. "bump Rand's WP from 6 to 8 and TR from 7 to 5"); don't
just say "the vector needs work".

## Files this skill touches

Read-only:
- `lib/philosophers.ts` — the corpus
- `scripts/check-philosopher-calibration.mjs` — the mechanical check
- `scripts/gen-philosophers.mjs` — for tradition baselines if a fix
  is needed
- `scripts/calibration-report.md` — generated by the mechanical run

Write (only when the user approves a fix):
- `scripts/gen-philosophers.mjs` ENTRIES array — if a per-entry
  override is the right fix
- `lib/philosophers.ts` — only via re-running the generator with
  `--apply`, never by direct hand-edit

## Output format

Always end the audit with this template, even when the verdict is
"everything looks fine":

```
## Calibration audit summary

- Reviewed: <N> entries flagged by the mechanical check
- Fine: <count>
- Suspicious: <count>
- Recommended action: [one of]
  - No changes needed
  - Adjust gen-philosophers.mjs: <specific entries + dim changes>
  - Defer for human review: <list>
```

## What NOT to do

- Don't run the mechanical check and then declare the corpus fine
  without doing the semantic review pass. The mechanics only flag
  geometric outliers; many real calibration bugs are semantic.
- Don't modify `lib/philosophers.ts` directly. The Wave 2 block is
  generator-owned (see the BEGIN/END sentinels). Fix the source
  (`gen-philosophers.mjs` ENTRIES) and re-run --apply.
- Don't propose "rewrite all vectors" as a fix. The corpus's density
  is a property of the 16-D model, not a bug.
- Don't change archetype thresholds in the mechanical check script
  to silence findings. The thresholds are calibrated against Wave 1;
  if they're firing on Wave 2 entries, those entries probably need
  attention.

## A note on archetype-margin density

Roughly 60% of the corpus has an archetype margin under 0.02. That's
not a calibration bug — it's a property of dense 16-D positive
vectors under cosine similarity. The model is most useful as a
spectrum positioning ("you score 8 on TV and 3 on MR") rather than
discrete classification ("you are a Pilgrim"). The archetype is a
useful label; the vector is the truth. Communicate this clearly to
the user when reporting the audit.
